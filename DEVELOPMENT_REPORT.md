## 🆕 新規実装ツール: naming-checker

### 概要
JavaScript/TypeScriptの変数・関数・クラス名の命名規則を自動検証するツールを新規開発しました。

### 実装した機能

#### 1. 検出パターン（5種類）
1. **snake_case違反検出**
   - 変数・関数でのスネークケース使用を検出
   - 例: `user_name` → 違反

2. **定数命名規則違反**
   - constが大文字スネークケースでないことを検出
   - 例: `const maxUsers` → 違反（`MAX_USERS`であるべき）

3. **PascalCase違反**
   - クラス名がパスカルケースでないことを検出
   - 例: `class userModel` → 違反

4. **camelCase違反**
   - 変数がキャメルケースでないことを検出
   - 例: `let UserName` → 違反

5. **変数名長違反**
   - 設定可能な最小/最大長チェック
   - デフォルト: 2文字以上、50文字以下

#### 2. 出力形式（3種類）

- **Console出力**: カラー表示で視認性向上
- **JSON出力**: CI/CD統合対応
- **CSV出力**: 分析・レポート作成用

#### 3. テストデータ

- `good_naming.js`: 正しい命名規則の例（70行）
- `bad_naming.js`: 50パターンの違反例
- `answer_key.json`: 期待される検出結果の詳細定義
- `run_test.sh`: 自動精度検証スクリプト

### 技術的な実装詳細

```bash
├── naming-checker          # メイン実行スクリプト (360行)

```

### 使用例

```bash
# 基本的な使用
./naming-checker ./src

# JSON出力でCI/CD統合
./naming-checker --output json ./src > report.json

# 統計情報付き実行
./naming-checker --stats ./project

# スネークケースを許可（柔軟な設定）
./naming-checker --allow-snake ./src
```

## 🔍 技術的課題と解決アプローチ

### 遭遇した主な課題

#### 1. Bashサブシェルでの変数スコープ問題
**問題**: プロセス置換 `< <(find ...)` を使用すると、whileループがサブシェルで実行され、親シェルの変数（カウンター等）が更新されない。

**解決策**: 
```bash
# 問題のあるパターン
while read file; do
    ((counter++))  # サブシェル内で実行、親に反映されない
done < <(find ...)

# 解決パターン
files=()
while read file; do
    files+=("$file")
done < <(find ...)

for file in "${files[@]}"; do
    ((counter++))  # 親シェルで実行、正しく反映される
done
```

#### 2. set -euo pipefail との相互作用
**問題**: common.shで設定された`set -euo pipefail`がスクリプト全体に適用され、想定外のエラー終了が発生。

**影響**:
- 未定義変数アクセスで即座に終了
- 算術演算の失敗で終了
- パイプ内のエラーが伝播

**対処**:
- 一時的に`set -o pipefail`のみに変更
- 変数の明示的な初期化
- エラーハンドリングの見直し

#### 3. カラーコードの不一致
**問題**: common.shは`NC`(No Color)を使用するが、ツールで`RESET`を参照していた。

**解決**: 
```bash
# 追加のカラー定義
BOLD='\033[1m'
RESET='\033[0m'
```

### デバッグで学んだ教訓

1. **段階的なデバッグの重要性**: 
   - 小さな単位でテスト
   - echo文による変数トレース
   - bash -x による実行トレース

2. **Bashの制限事項**:
   - 複雑なロジックにはBashは不向き
   - 大規模ツールにはNode.js/Python推奨

3. **set -e の慎重な使用**:
   - デバッグ時は無効化を検討
   - エラーハンドリングを明示的に

1. **複数ファイルスキャンの不安定性**
   - 状況: 大規模プロジェクトで一部のファイルのみ処理される
   - 原因: エラーハンドリングとサブシェル問題の未解決部分
   - 影響: 中〜大規模プロジェクトでの実用性に制限

2. **検出精度**
   - 目標: 80-90%の検出率
   - 現状: 単一ファイルでは機能するが、複数ファイルで問題
   - 改善が必要な領域

3. **パフォーマンス**
   - 大規模プロジェクト（1000+ファイル）での未テスト
   - 最適化の余地あり
